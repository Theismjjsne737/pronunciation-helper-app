--- Document: Introduction to RAG ---
Retrieval-Augmented Generation (RAG) is a technique that combines information retrieval with language model generation. Instead of relying solely on the model's trained knowledge, RAG first retrieves relevant documents from a knowledge base, then uses those documents as context to generate an answer. This makes responses more accurate, up-to-date, and grounded in specific sources.

--- Document: How Vector Databases Work ---
A vector database stores data as high-dimensional numerical vectors (embeddings). When you embed text using a model like SentenceTransformer, each chunk of text becomes a vector that captures its semantic meaning. Similar texts produce similar vectors. During retrieval, a query is embedded and the database finds the nearest vectors using cosine similarity or dot product, returning the most semantically relevant chunks.

--- Document: Sentence Transformers ---
Sentence Transformers are a class of models that map text to dense vector embeddings. They are trained using siamese and triplet networks on large datasets so that semantically similar sentences are close in the embedding space. The model "all-MiniLM-L6-v2" is a popular lightweight choice that balances speed and quality. These embeddings are suitable for semantic search, clustering, and retrieval tasks.

--- Document: ChromaDB Overview ---
ChromaDB is an open-source embedding database designed for AI applications. It supports storing documents with their embeddings and metadata, performing similarity search, and persisting data to disk. ChromaDB can run fully in-process (no separate server needed), making it ideal for local development. It integrates with popular embedding models and LLMs and supports filtering metadata during retrieval.

--- Document: Text Chunking Strategies ---
Chunking is the process of splitting documents into smaller pieces before embedding. Good chunking is critical for RAG quality. Common strategies include fixed-size chunking (splitting every N characters or tokens), sentence-level chunking (splitting on sentence boundaries), and recursive chunking (trying multiple separators in order). Overlap between chunks helps preserve context at boundaries. Typical chunk sizes range from 200 to 1000 tokens depending on the use case.

--- Document: LLM Context Windows ---
Language models have a maximum context window — the number of tokens they can process at once. When using RAG, retrieved chunks must fit within this window alongside the question and system prompt. Claude Sonnet 4.6 supports a 200K token context window, which is large enough to include many retrieved chunks. Despite this, it is still best practice to retrieve only the most relevant 3–5 chunks to keep costs low and answers focused.

--- Document: Cosine Similarity ---
Cosine similarity measures the angle between two vectors in high-dimensional space. A value of 1.0 means the vectors point in the same direction (identical semantics), 0 means orthogonal (unrelated), and -1 means opposite. In embedding-based retrieval, cosine similarity is the standard metric because it is invariant to vector magnitude — only direction matters, not the absolute length of the embedding.

--- Document: Citations in AI Answers ---
Citing sources in AI-generated answers improves trust and allows users to verify information. In a RAG system, each retrieved chunk should carry metadata (document name, page number, URL, etc.) that the LLM can reference in its response. A good citation format includes the source name and, where possible, the exact passage used. This prevents hallucination by anchoring claims to specific retrieved text.
